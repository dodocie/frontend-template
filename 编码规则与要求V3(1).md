# Trae Project Specific Rules (项目专属规范)

## 1. 技术栈与环境规范 (Tech Stack)

- **Runtime**: Node.js v20+。
- **Frontend Framework**:
  - **Vue 3**: 必须使用 `Composition API` + `<script setup lang="ts">`。
  - **React**: 必须使用 `Functional Components` + `Hooks` + `TypeScript`。
- **Routing (路由规范)**:
  - **严禁**使用响应式变量（如 `v-if="page === 'home'"`）手动模拟页面切换。
  - **必须**使用官方路由库：Vue 使用 `vue-router`，React 使用 `react-router-dom`。
- **State Management (状态管理)**:
  - **Vue**: 必须使用 **Pinia** 处理全局/跨组件状态。
  - **React**: 优先使用 **Zustand** 或 **Jotai**（根据项目轻重程度选择）。
  - **复杂逻辑控制 (State Machine)**: 涉及多步表单、复杂流程切换或大量条件分支时，**结合业务需求可考虑使用 XState** 管理状态机。严禁在模板或组件中堆砌大量的 `if...else` 或 `switch` 渲染逻辑。
- **Type System**: **必须全面使用 TypeScript**。开启 `strict` 模式，严禁滥用 `any`。

## 2. 组件库与样式 (UI & Styling)

- **Styling**: **必须使用 Tailwind CSS**。
- **Component Libraries (组件库优先策略)**:
  - **原则**：优先寻找现有组件，仅在无法满足需求时才手写 CSS。
  - **Vue**: PC端使用 **Element Plus**，移动端使用 **Vant UI**。
  - **React**: 优先使用 **Mantine** (覆盖 PC 与移动端)，或使用 **Radix UI** (搭配 **Shadcn/UI**)。
- **外链资源引用规范**：
  - **禁止行为**：严禁在 CSS/SCSS 中通过 `@import url()` 引入外链资源（如字体、样式文件）。
  - **合规加载方式（按优先级排序）**：
    - 将外链资源（字体、样式文件等）下载到本地后引入。
    - 使用非阻塞 <link> 引入。

## 3. 代码质量与 Lint (Code Quality)

- **代码检查**: **必须配置 ESLint + TypeScript-ESLint**。
- **风格统一**: 必须集成 **Prettier**。
- **数据源统一 (Centralized Data)**:
  - **禁止**在组件内散落 `fallback` 数据（如 `const list = props.data || []`）。
  - **Mock 数据中心化**：所有 Mock 数据必须存放在 `src/mocks/` 或 `src/api/mock/` 目录下，并导出为统一变量或函数。禁止在函数/方法内通过硬编码常量补全测试数据，测试数据缺失时应抛出异常而非静默补全。
- **DOM 操作限制**: 严禁使用原生 JavaScript 直接操作 DOM。必须完全依赖声明式渲染（Vue 的响应式系统或 React 的 State）。
- **单一职责函数拆分**：
  - 所有大功能模块必须拆解为单一职责的小函数，按职责划分为「数据处理」「模型操作」「文件操作」等类别，禁止一个函数包含多类职责逻辑。
  - 数据处理函数：仅负责数据的转换、校验、格式化、聚合等纯数据操作，禁止包含模型状态修改、文件读写等逻辑。
  - 模型操作函数：仅负责状态管理模型（Pinia/Zustand/Jotai/XState）的状态读写、行为触发，禁止包含数据处理、文件操作等逻辑。
  - 文件操作函数：仅负责文件的读取、写入、删除、上传 / 下载等操作，禁止包含数据处理、模型状态修改等逻辑。
  - 函数命名需体现具体职责（如 formatUserFormData/updateUserInfoStore/writeUserConfigFile），禁止使用模糊命名（如 handleUserInfo/doUserThings）。
  - 拆分后的函数需控制行数，单个函数代码行数超过 50 行需进一步确认职责是否越界，是否需要进一步拆解。
- **日志打印规范**：
  - 禁止无差别打印大量日志（如循环内逐行打印、全量打印接口返回数据、重复打印相同状态信息），避免消耗运行时资源、拖慢接口 / 页面响应速度。
  - 日志内容规范：所有保留的日志必须包含「模块名 + 函数名 + 关键上下文 + 核心信息」。
  - 日志工具统一：
    - 所有日志输出必须使用统一封装的日志工具（文档链接：待补充）。
    - 日志工具支持通过配置实现日志信息过滤，便于调试时精准定位指定模块日志、屏蔽无关日志（配置方式待补充）。

## 4. 主题与全局配置 (Theming)

- **主题配置**:
  - 必须有全局主题管理机制。
  - Tailwind 变量（颜色、间距等）必须在 `tailwind.config.js` 中定义。
  - 组件库的主题（如 Mantine 的 `MantineProvider` 或 Element Plus 的 CSS Variables）必须统一通过配置文件管理。

## 5. 项目目录地图与文档约束 (Documentation)

- **文档查阅与更新**:
  - 修改需求前必读 `docs/`。需求变更后必须同步更新，**严禁**创建 `v2`, `new` 等命名后缀的重复文档。
- **大型功能计划 (Plan 文档)**:
  - 复杂模块开发前，必须在 `docs/plans/` 下建立 `.md` 计划。
  - 任务使用 `[ ]` 列表，**每完成一个子项必须更新为** **`[x]`**。
- **操作脚本化 (Scripts)**:
  - 所有的重复性任务（构建、部署、数据清洗、Lint 检查）必须编写 Node.js 或 Shell 脚本放入 `scripts/` 目录。
  - 对话中通过 `npm run <script>` 形式调用。

## 6. AI 行为准则

- **自说明代码**: 变量/函数名需具备高度语义化，复杂逻辑必须附带 TypeScript 类型注释或 JSDoc。
- **模块化思维**: 优先考虑逻辑抽离（Vue Hooks / React Hooks）以及组件复用。
- **防御性编程**: 接口定义必须清晰，并处理好 Loading 和 Error 状态。
