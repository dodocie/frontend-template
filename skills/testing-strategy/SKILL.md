# Testing Strategy Skill

This skill enforces comprehensive testing practices.

## 1. 测试框架选择

*   **单元测试**: Vue 使用 Vitest，React 使用 Jest 或 Vitest。

## 2. 测试覆盖率

*   **最低要求**: 没要求，根据项目需求判断，非必要不写单元测试。
*   **关键路径**: 核心业务逻辑必须100%覆盖。

## 3. 测试策略

*   **TDD**: 鼓励采用测试驱动开发。
*   **Mock 策略**: 合理使用 Mock 避免外部依赖。

## 4. 单元测试覆盖范围指导

### ✅ 强烈建议写单元测试

#### 类型示例
*   **工具函数/纯函数**
    *   格式化函数、加解密、数据转换、深拷贝
*   **复杂业务逻辑**
    *   价格计算、权限校验、状态机、表单校验规则
*   **自定义 Hooks**
    *   包含副作用和状态逻辑的 React Hooks
*   **核心状态管理**
    *   Redux/Store 中的 reducer、selector、action 逻辑
*   **API 请求封装**
    *   请求参数构建、响应数据转换、错误处理逻辑

### ⚠️ 视情况决定

#### 类型判断依据
*   **UI 组件（有逻辑）**
    *   如果组件包含条件渲染、事件回调、状态变更，建议测试；如果只是纯展示组件，可酌情略过
*   **公共组件库**
    *   如果组件会被多个项目复用，需要较高测试覆盖率
*   **工具函数（简单）**
    *   如果函数过于简单（如 `const add = (a,b)=>a+b`），测试收益低，可略过

### ❌ 禁止写单元测试

#### 类型替代方案
*   **纯样式/布局**
    *   视觉回归测试、人工审查
*   **频繁变动的原型代码**
    *   等稳定后再补
*   **简单的事件绑定**
    *   集成测试覆盖
*   **第三方库调用（无封装逻辑）**
    *   信任库本身
*   **配置类文件**
    *   不需要测试
